# 0026 — 서비스 레지스트리(provider 귀속) + q2svc 단일 소스

- 상태: Accepted — 2026-07-14 (도메인 그릴링 — 데이터 실측 기반 2층 분리)
- 관련: [[0025-wordpress-headless-cms.md|ADR-0025]](CMS 계층) · CONTEXT.md '서비스'/'개념 카드'
- 코드: `infra/wp/mu-plugins/quizdeck-content/`(fields·cpt·save·rest·query) — **앱 무변경**

## 맥락

"개념/서비스 카드는 시험이 아니라 aws 자체에 귀속 아닌가?"(사용자, 2026-07-14). 라이브
데이터 전수 측정으로 판정했다:

- SAA(54)·SAP(174) 교집합은 라벨 기준 27뿐이고, SAA 전용 27개의 다수는 서비스가 아니라
  **비교·묶음·전략 학습 카드**("ALB vs NLB"·"DR 전략")다.
- 교집합 27쌍의 교육 필드(deff·key·when·trap·vs)는 **27/27 전부 분화** — 시험 눈높이로
  따로 쓰인 텍스트라 provider 로 합치면 편집 판단 27건이 필요하고 눈높이가 손실된다.
- 반면 **정체성 층은 provider 의 것**인데 실체가 없었다: 카드의 표시 라벨이 곧 키(Topic
  에서 금지한 라벨-as-키), 아이콘은 시험마다 재저작(SAP 174 · SAA 0), cat 표기도 어긋남.
- 추가 실측: 카드의 rel/reln(관련 문항 — 저장된 파생 데이터)과 exam 의 q2svc 가 **이미
  드리프트**(SAP 카드 37건 reln 불일치·34건 rel⊄q2svc, "리전/AZ" reln=83 vs q2svc 0).

## 결정

1. **2층 분리(CONTEXT.md 확정)** — Service = provider 귀속 정체성(id·이름·약어·아이콘·
   분류), 개념 카드 = Exam 귀속 학습 노트가 Service 를 0..n 참조(비교 카드 n개·전략 카드
   0개). 같은 서비스라도 카드는 시험마다 다른 것이 정당하다.
2. **qd_service CPT**(qd-services REST) — (provider, service_id) 유일성은 파생 복합키
   `qd_service_key` 로 게이트(exam_key 패턴), id 는 `^[a-z0-9-]+$`(언어 무관 안정 키).
   카드는 `qd_service_ids`(JSON 배열)로 참조 — 게시 게이트가 레지스트리 존재를 검증.
3. **q2svc = 문항↔카드 관계의 단일 소스.** rel/reln 저장 폐기 — REST 투영이 q2svc
   역인덱스에서 파생(rel = 오름차순 ≤40 — 기존 데이터 max 40, reln = 총수). 드리프트가
   구조적으로 불가능해진다.
4. **파생은 REST 투영(PHP)에서** — 봉투 계약이 구 저장 필드와 동일해 **앱·wp-client
   무변경**, 앱/WP 배포 결합 없음. exam 의 svc_icons 는 "레지스트리 파생(카드→첫 참조
   서비스의 아이콘) 오버레이 + 레거시 블롭 폴백" — 서비스에서 아이콘을 한 번 고치면 참조
   시험 전부에 반영. 파생 아이콘은 transient 캐시(콘텐츠 저장 시 전량 무효화 — 시험 2~3개
   규모에 선택적 무효화는 과설계).
5. **시딩은 무결한 것만 기계로** — "Amazon |AWS " 접두 + sanitize_title 이 ascii 인 카드만
   서비스 생성·연결(일회성 스크립트, 멱등: draft 삽입→meta→publish 승격으로 save_post
   게이트 함정 회피). 접두 없는 실서비스("Transit Gateway"·"S3 Glacier")와 묶음 카드
   구분은 편집 판단 — admin 에서 점진 연결(레거시 아이콘 폴백이 있어 무회귀).
6. **병합 = 합집합** — 구 rel 을 q2svc 에 편입(rel 도 이관 검증을 거친 데이터 — 수동 백필
   83건은 비현실, 합집합은 하한 보장) 후 rel/reln meta 삭제.

## 애던덤 — qn 불변 게이트 (2026-07-14)

데이터 모델 조사의 후속 결정: 학습자 데이터 전부(이력·오답·컬렉션·주석)가 `(examKey, qn)`
으로 문항을 참조하는데 WP 에는 유일성 게이트만 있고 **불변성** 보호가 없었다 — 게시 중
qn 변경·삭제가 학습 기록을 조용히 고아화한다.

- **셰도우 메타 `qd_qn_published`** = 마지막 게시 시점 qn. 게시 상태에서 현재 qn 과 다르면
  게시 거부(draft 강등 + 사유 — 기존 게이트 규율). 폼·REST 양 경로 동일.
- **탈출구 = 의도적 2단계**: 사용자가 초안·휴지통으로 내리면 셰도우 해제(transition 훅)
  → 변경 → 재게시. 단 **검증 실패의 자동 강등은 해제하지 않는다**(qd_demoting 플래그) —
  게이트가 스스로를 우회하게 두지 않는 것이 설계의 핵심.
- 삭제는 막지 않는다 — 휴지통 이동 시 경고 알림(학습 기록 참조 고지)으로 충분(1인 운영).
- 기존 게시 문항은 셰도우가 없어 첫 재저장 전까지 무방비 → 배포 시 백필(787건) 1회.
- 검증(로컬): 게시 중 변경→강등+셰도우 유지, 복원 재게시→통과, 초안 경유 변경→통과+새
  셰도우, 휴지통→해제+경고. 5 시나리오 전부 실측.

## 기각 대안 (재제안 방지)

- **카드 전체 provider 귀속** — 데이터가 반박(전 카드 분화 + 비서비스 카드 존재).
- **rel 을 관계 소스로** — 서비스맵(q2svc)이 카드 편집에 종속되고 다중 참조 카드와 충돌.
- **현상 유지 + 정합 검사만** — 이중 저작 부담·드리프트 원인 존치.
- **파생을 앱(wp-client)에서** — 가능하나 앱/WP 배포 결합 + 두 소비 경로 중복 구현.
