# 0009 — 아이콘 시스템: react-icons(Lucide UI + simple-icons 브랜드), 점진 적용

Status: accepted — 파일럿(auth 영역) 2026-06-30, 점진 마이그레이션 진행 중

## 맥락

아이콘 라이브러리가 없다. 아이콘 역할을 **이모지**(🔑 패스키·✉️ 메일·⚠️·🎉…)와 **텍스트 글리프**(닫기 `✕` ~96곳, `←`/`→` 등)가 대신하며 `components/`·`app/` 전반 ~90곳에 흩어져 있다. 문제:

- **일관성 없음** — 이모지는 OS·폰트마다 렌더가 달라(특히 ✕·← 같은 기호) 모양·굵기가 들쭉날쭉.
- **테마 비대응** — 색·크기를 CSS-var 토큰(`--muted`·`--fg`…)으로 제어 못 한다. 라이트/다크에서 색이 따라오지 않는다.
- **접근성 모호** — 장식인지 의미인지 구분이 없다(스크린리더가 이모지를 읽어버리거나, 아이콘-only 버튼에 라벨이 없다).
- **광학 정렬** — 이모지는 baseline·메트릭이 제각각이라 텍스트와 수직 정렬이 어긋난다.

소셜 로그인([[0003-auth-and-progress-sync.md|#9]])에서 **브랜드 로고**(GitHub·Google·Naver)가 필요해지며 이 문제가 표면화됐다 — 텍스트만으론 식별성이 약하다.

[[0008-ui-interaction-and-components.md|ADR-0008]] 결정 2는 "**컴포넌트는 무의존 직접 구축**(cva·Radix·shadcn 없음)"을 못박았다. 아이콘 라이브러리는 이 철학과 충돌하는가? — **아니다**(결정 2에서 구분).

## 결정

1. **공용 아이콘 라이브러리 = `react-icons`.** UI 아이콘은 `react-icons/lu`(Lucide 라인 셋), 브랜드 로고는 `react-icons/si`(simple-icons — GitHub·Google·Naver). 한 패키지로 UI+브랜드를 통일 소싱한다. 서브경로 import(`react-icons/lu`·`react-icons/si`)라 **쓰는 아이콘만 번들**된다(트리셰이킹).

2. **이는 ADR-0008 결정 2(무의존 컴포넌트)와 모순이 아니다.** react-icons 는 동작·관습·스타일을 들이는 **컴포넌트 프레임워크**(cva 의 variant 문법, Radix 의 `data-state`·focus-trap, shadcn 의 `cn`)가 아니라 **SVG path 데이터를 React 로 감싼 에셋**이다. "직접 구축" 대안은 수십 개 SVG path(특히 **변하는** 브랜드 로고)를 손으로 유지하는 것 — 학습가치 0, 순수 유지보수 부채다. 따라서 아이콘은 **데이터 의존성**으로 채택하며, ADR-0008 이 거부한 **프레임워크 의존성**과 범주가 다르다. (Lucide 글리프 자체는 ADR-0008 이 직접 만들었어도 골랐을 라인 스타일 — react-icons 는 소싱 수단일 뿐.)

3. **아이콘은 토큰 테마에 대응한다.** 색은 `currentColor`(=주변 텍스트 색, `--muted`/`--fg` 추종), 크기는 `em`/`className`(`size-4` 등)으로. 장식 아이콘은 `aria-hidden`, 의미는 라벨·`aria-label`이 전한다 — **아이콘-only 버튼엔 `aria-label` 필수**(ADR-0008 의 a11y floor 와 같은 결).

4. **브랜드 로고는 단색**(simple-icons 제약). **Google 4색 G 포기를 수용**한다(통일 소싱·일관성 우선). 중립(outline) 버튼 안에서는 **세 브랜드 마크를 모두 `currentColor`로** 둔다 — 한 로고만 브랜드색(예: Naver 초록)이면 나머지와 어긋나 시각적으로 튄다. 브랜드색은 그 색이 면(filled) UI 의 주역인 맥락(예: 브랜드색 배경 버튼)에서만. 특정 로고에 4색이 꼭 필요하면 **그 로고만 인라인 SVG 예외**를 허용한다(규칙이 아니라 예외).

5. **마이그레이션은 공존·점진**(ADR-0008 결정 4 와 같은 결). **auth 영역**(소셜·패스키 버튼, 로그인 모달 close)을 **파일럿**으로 패턴을 확립하고, 나머지 ~90곳(✕·이모지)은 "**건드리는 코드부터**" 점진 교체한다. 65곳 빅뱅을 피한 ADR-0008 의 검증된 패턴을 따른다.

## 왜

- 이모지 → 벡터 아이콘은 **일관성·테마대응·접근성·광학정렬**을 한 번에 얻는다.
- react-icons 한 패키지로 **UI(Lucide) + 브랜드(simple-icons)를 통일 소싱** — import 지점 하나, 트리셰이킹으로 번들 영향 최소.
- 아이콘은 **데이터**라, 프레임워크 의존을 피하려는 ADR-0008 의 minimal-deps 철학과 충돌하지 않는다. "미리 사지 않음"(lib/email.ts)은 **추상화**를 미리 사지 말라는 것이지, 변하는 에셋을 손으로 베끼라는 것이 아니다.
- **점진 적용**은 빅뱅의 고-churn·회귀 위험을 피한다.

## 고려한 대안 (재제안 방지)

- **`lucide-react`(직접 의존)** — UI 엔 깔끔하나 **브랜드 로고가 없다**(lucide 는 트레이드마크 이유로 브랜드 제거). 브랜드는 별도 인라인 SVG 가 필요해 소싱이 둘로 갈린다. react-icons 는 `Lu*`(동일 Lucide 글리프) + `Si*`(브랜드)를 **한 소스**로 줘 통일성이 높다. 기각.
- **인라인 SVG 만(의존성 0)** — 의존성은 0 이나 브랜드+UI 글리프를 손으로 유지하는 부채, 세트 일관성·광학 정렬을 직접 관리. 아이콘은 데이터라 라이브러리가 합리적. (단 4색 브랜드 예외는 인라인 — 결정 4.)
- **`@heroicons/react`** — Tailwind 궁합은 좋으나 아이콘 수가 적고(~300) 브랜드 로고가 없다.
- **브랜드 4색 유지(인라인 Google G) + UI 만 라이브러리** — 혼합 소싱. 통일성을 위해 단색 수용으로 기각(필요 시 특정 로고만 인라인 예외 — 결정 4).
- **react-icons 빅뱅(90곳 일괄)** — 완전 일관성이나 고-churn·회귀·리뷰 부담. base 가 affordance 를 이미 깐 ADR-0008 처럼, 점진이 안전(결정 5).
