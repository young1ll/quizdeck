🗄️ 리소스 토폴로지 — 한 호스트, 두 tier

하나의 Synology 호스트(zero)가 VMM으로 두 VM을 띄웁니다 — app tier(k3s)와 data tier(postgres)를 분리. 둘은 OVS LAN에서 게스트↔게스트로만 통하고, postgres는 어디서도 인터넷에 노출되지 않습니다.

🐘 데이터 tier — 격리된 postgres (db-home)

postgres를 k3s 클러스터 밖의 별도 VM에 둬 RDS의 private-subnet 격리를 실습합니다. DB 수명을 k3s 재설치와 분리하고, 다층 방어(방화벽 + pg_hba)로 오직 k3s-home만 접근하게 합니다.

격리 (다층 방어)

  • ufw default-deny · 5432는 192.168.68.55/32(k3s-home)에서만. SSH(22)는 LAN+타넷만.
  • pg_hba — quizdeck role은 k3s-home에서만(scram-sha-256). listen=localhost,.52.
  • 인터넷 차단 — 라우터 포트포워딩 0 + ufw로 호스트·인터넷·그 외 타넷 모두 거부.
  • 검증: k3s-home→5432 OPEN, Mac·호스트→5432 차단 확인.

스키마 · 시크릿

  • better-auth 테이블: user·session·account·verification·jwks (라이브러리 마이그레이션).
  • 접속정보는 k8s Secret db-credentials(ns quizdeck, git 밖) — 앱이 secretKeyRef로 주입.
  • 프로비저닝: VMM + cidata autoinstall seed(k3s-home과 동일 패턴) → 멱등 스크립트(provision-postgres.sh 등).
  • 진도(progress) 테이블은 V2(LWW 동기화) 때.

🔑 인증 — better-auth (in-app)

무거운 IdP 대신 better-auth를 standalone 서버의 Route Handler(app/api/auth/*)에 in-app으로 둡니다. 같은 오리진 쿠키 세션 + JWKS 노출로 미래 NestJS pod가 JWT를 검증할 경로를 열어 둡니다. 구현 완료 · 배포 직전 (PR #14)

이번 슬라이스 (V1)

  • 이메일+비밀번호 가입·로그인·로그아웃. 익명 사용은 그대로(로그인은 선택).
  • 세션 = 같은 오리진 쿠키(Secure·HttpOnly·SameSite) — 새로고침·탭 재오픈에도 유지.
  • JWKS GET /api/auth/jwks 노출(지금 소비자 없음, IdP-lite 경로).
  • 표준 보호: 비밀번호 해시(scrypt)·CSRF(origin 체크)·rate-limit. 시크릿 quizdeck-auth(git 밖).

후속

  • V2 — 진도 동기화: local-first composite ProgressStore(localStorage + RemoteApi, LWW). ADR-0001 seam 활용.
  • V4/V5 — 소셜 로그인(GitHub·Google·Naver) · 패스키(WebAuthn). 같은 토대 위에 얹음.
  • SMTP 미구성 → 이메일 검증 OFF(가입 즉시 로그인).

📶 진화 로드맵 — 층(Layer)

각 상위 층은 독립적으로 가치 있고, 가치 < 비용 지점에서 멈춥니다. 플랫폼은 워크로드 위에서 진화합니다.
